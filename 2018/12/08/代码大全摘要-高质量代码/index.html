代码大全摘要-高质量代码

Dec 8, 2018

构建中的设计

设计中的挑战

  • 设计是个险恶的问题（把问题解决一遍能够明确定义它，然后再次解决它）
  • 设计是个了无章法的过程
  • 设计就是确定取舍和调整顺序的过程
  • 设计受到诸多限制
  • 设计是不确定的
  • 设计是一个启发式过程（过程中总会有试验和犯错误）
  • 设计是自然而然形成的

关键的设计概念

管理复杂度

  • 把任何人在同一时间需要处理的本质复杂度的量减到最小
  • 不要让偶然性的复杂度无谓地快速增长

理想的设计特征

  • 最小的复杂度
  • 易于维护
  • 松散耦合
  • 可扩展性
  • 可重用性
  • 高扇入
  • 低扇出
  • 可移植性
  • 精简性
  • 层次性
  • 标准技术

设计的层次

软件系统 -> 子系统和包 -> 包中的类 -> 类中的数据和子程序 -> 子程序内部
（子系统依赖关系应该是有向无环图）

设计构造块（启发式方法）

形成一致的抽象

抽象是能让你在关注某一概念的同时可以放心忽略其中一些细节

封装实现细节

封装帮助你管理复杂度的方法是不让你看到那些复杂度

信息隐藏

隐藏起来的可能是某个易变的区域，在这个区域中发生的错误不会给程序其余部分带来太大损失
对系统非重大的改动可能只会影响到某个类中的几个子程序

秘密分为两类

  • 隐藏复杂度，这样你就不用再去关注他
  • 隐藏变化源，发生变化时，其影响就能被限制在局部范围内

信息隐藏的障碍

  • 信息过度分散
  • 循环依赖（无法单独测试）
  • 类内数据误认为全局数据
  • 可以察觉的性能损耗

找出容易改变的区域

  • 业务规则
  • 对硬件的依赖性
  • 输入和输出
  • 非标准的语言特性
  • 困难的设计区域和构建区域
  • 状态变量
  • 数据量的限制

查阅常用的设计模式

设计模式集成了众多现成的解决方案

使用设计模式的优势

  • 设计模式通过提供现成的抽象来减少复杂度
  • 把常见解决方案的细节予以制度化来减少出错
  • 提供多种设计方案而带来启发性的价值
  • 简化交流

其他的启发式方法

  • 高内聚性
  • 构造分层结构
  • 严格描述类契约
  • 分配职责
  • 为测试而设计
  • 避免失误
  • 有意识地选择绑定时间
  • 创建中央控制点
  • 考虑使用蛮力突破
  • 画一个图（一图顶千语）
  • 保持设计的模块化

设计实践

  • 迭代 — 可以从高层次或低层次去审视问题，从不同角度
  • 分治法 — 把程序分解为不同的关注区域，然后分别解决每一个区域
  • 自上而下 — 从一般性问题出发，把问题分解成可控的部分，一种分解策略
  • 自下而上 — 从可控的部分出发，去构造一个通用的方案，一种合成策略
  • 建立试验性原型
  • 合作设计

要做多少设计才够

最大的设计问题来自于那些误认为很简单的，而没有做出足够设计的区域

记录你的设计成果（适用于小型项目）

  • 把设计文档插入到代码中
  • 用wiki来记录设计讨论和决策
  • 写总结邮件
  • 使用照片方式记录白板
  • 在适当的细节层创建uml图

可以工作的类

ADT（抽象数据类型）

一些数据及对这些数据所进行的操作集合

使用ADT的益处

  • 可以隐藏实现细节
  • 改动不会影响到整个程序，也是细节隐藏的好处
  • 让接口能提供更多信息
  • 更容易提高性能
  • 让程序的正确性更显而易见
  • 程序更具有自我说明性
  • 无须在程序内到处传递数据
  • 你可以像在现实世界中那样操作实体

ADT示例

  • 把底层的数据结构包装成ADT
  • 把像文件这样的常用对象当成ADT
  • 简单的事物也当成ADT
  • 不要让ADT依赖与存储介质

ADT构成了类这一概念的基础，加上了继承和多态

良好的类接口

好的抽象

  • 类的接口应该展现一致的抽象层次（四不像的东西需要重新设计）
  • 一定要理解类所实现的抽象是什么
  • 提供成对的服务（符合自然常识）
  • 把不相关的信息转移到其他类中
  • 尽可能让接口可编程，而不是表达语义
  • 防止在修改时破坏接口的抽象
  • 不要添加与接口抽象不一致的公用成员
  • 同时考虑抽象性和内聚性

良好的封装

  • 尽可能地限制类和成员的可访问性
  • 不要公开暴露成员数据
  • 避免把私用的实现细节放入类的接口中（不要暴露细节）
  • 不要对类的使用者做出任何假设
  • 让阅读代码比编写代码更方便
  • 要格外警惕从语义上破坏封装性
  • 留意过于紧密的耦合关系

有关设计与实现

包含关系（“有一个”的关系）

  • 优先考虑使用包含，万不得已时使用继承
  • 警惕数据成员超过7个数据成员的类

继承关系（“是一个”的关系）

  • 确保只继承需要继承的部分
  • 不要覆盖一个不可覆盖的成员函数
  • 把共用的接口、数据及操作放到继承树尽可能高的位置
  • 只有一个实例的类是值得怀疑的
  • 只有一个派生类的基类也是值得怀疑的 — 过度设计
  • 派生覆盖某个子程序后，在其中没做任何操作，这种情况也是值得怀疑的 — 改变语义
  • 避免让继承体系过深 — 管理复杂度
  • 尽量使用多态，避免大量的类型检查
  • 让所有数据都是private（拥有最强的可控性）

包含与继承的选择

  • 如果多个类共享数据而非行为，应该创建这些类可以包含的共用对象
  • 如果多个类共享行为而非数据，应该让它们从共同的基类继承而来，并在基类中定义共用的子程序
  • 如果多个类共享行为也共享数据，应该让它们从共同的基类继承而来，并在基类中定义共用的子程序和数据
  • 当你想基类控制接口时，使用继承；自己控制接口时使用包含

成员函数与数据成员

  • 让类中子程序的数量尽可能少
  • 禁止隐式地产生你不需要的成员函数与运算符
  • 减少类所调用的不同子程序的数量
  • 对其他类的子程序的间接调用要尽可能少
  • 一般来说，应尽量减少类和类之间相互合作的范围

构造函数

  • 如果可能，应该在所有的构造函数中初始化所有的数据成员
  • 用私用构造函数来强制实现单例
  • 优先深拷贝，除非论证可行才采用浅拷贝

创建类的原因

  • 为现实世界中的对象建模
  • 为抽象的对象建模
  • 降低复杂性
  • 隔离复杂度
  • 隐藏实现细节
  • 限制变动的范围
  • 隐藏全局数据
  • 让参数传递更顺畅
  • 建立中心控制点
  • 让代码更易于重用
  • 把相关操作包装到一起
  • 实现某种特定的重构

应该避免的类

  • 避免创建万能类
  • 消除无关紧要的类
  • 避免用动词命名的类

高质量的子程序

创建子程序的正当理由

  • 降低复杂度
  • 引入中间、易懂的抽象
  • 避免代码重复
  • 支持子类化
  • 隐藏顺序
  • 提高可移植性
  • 简化复杂的布尔判断
  • 降低性能优化成本

好的子程序名字

  • 描述子程序所做的所有事情
  • 避免使用无意义的、模糊或表达不清的动词
  • 不要仅通过数字来形成不同的子程序名字
  • 根据需要确定子程序名字的长度
  • 给函数命名时要对返回值有所描述
  • 给过程起名时使用强烈语气的动词加宾语的形式
  • 准确使用对仗词
  • 为常用操作确定命名规范

子程序可以写多长

建议不要超过200行，长度过长会影响子程序的可读性

如何使用子程序参数

  • 按照输入-修改-输出的顺序排列参数
  • 使用所有的参数
  • 把状态或者出错变量放在最后
  • 不要把子程序的参数用做工作变量
  • 在接口中对参数的假定加以说明
  • 参数个数限制在7个之内
  • 确保实际参数与形式参数相匹配

使用函数时需要考虑的问题

何时使用函数，何时使用过程

如果一个子程序主要用途返回其名字所指明的返回值，那么就应用使用函数，否则使用过程

设置函数的返回值

  • 检查所有可能的返回路径
  • 不要返回指向局部数据的引用和指针

防御式编程

保护程序免遭非法输入数据的破坏

  • 检查所有来源于外部的数据值
  • 检查子程序所有输入参数的值
  • 决定如何处理错误的输入数据

断言

开发期间使用的，让程序在运行时进行自检的代码

使用断言的指导建议

  • 用错误处理代码来处理预期会发生的状况，用断言处理绝对不应该发生的状况
  • 避免把需要执行的代码放到断言中
  • 用断言来注解并验证前条件和后条件
  • 对于高健壮性的代码，先处理断言再处理错误

错误处理技术

  • 返回中立值
  • 换用下一个正确的数据
  • 返回与前次相同的数据
  • 返回最接近的合法值
  • 把错误信息记录到日志文件中
  • 返回一个错误码
  • 调用错误处理子程序或对象
  • 当错误发生时显示错误信息
  • 关闭程序

健壮性与正确性

  • 正确性意味永不返回不准确的结果
  • 人身安全攸关的软件：倾向于正确性而非健壮性
  • 消费类应用软件：更注重健壮性而非正确性

高层次设计对错误处理方式的影响

  • 确定一种通用的处理错误参数的方法
  • 确定好方法后确保始终如一地贯彻这一方法

异常

异常是把代码中的错误和异常事件传递给调用方的一种手段

异常使用建议

  • 用异常通知程序的其他部分，发生了不可忽略的错误
  • 只有真正例外的情况才会抛出异常
  • 不能用异常来推卸责任（可以在局部处理就应该处理掉它）
  • 在恰当的抽象层次抛出异常
  • 避免使用空的catch语句
  • 考虑所用函数库可能抛出的异常
  • 考虑创建一个集中的异常报告机制
  • 把项目中的异常的使用标准化

隔离程序（一种容错策略）

在输入数据时将其转换为恰当的类型

隔离与断言的关系

隔栏外部的程序应该使用错误处理技术，在隔离内部的程序应该使用断言技术

辅助调试的代码

在开发期间牺牲一些速度和对资源的使用，来换取一些可以让开发更顺畅的内置工具

应尽早引入辅助调试的代码

对你的帮助最大化

采用进攻式编程（有时候最好的防守正是大胆的进攻）

  • 让问题引起的麻烦越大越好，避免忽略问题
  • 完全填充分配到的内存
  • 完全填充已分配到的所有文件或流
  • 确保每个case语句中的default能产生严重错误，避免忽略
  • 删除一个对象时把它填满垃圾数据

确定在产品代码中该保留多少防御代码

  • 保留那些检查重要错误的代码
  • 去掉检查细微错误的代码
  • 去掉可以导致程序硬性崩溃的代码
  • 保留可以让程序稳妥地崩溃的代码
  • 为你的技术支持人员记录错误信息
  • 确认留在代码中的错误信息是友好的

对防御式编程采取防御的姿态

过度防御会导致代码臃肿与性能下降，因地制宜的调整你进行防御式编程的优先级

伪代码编程过程

创建类和子程序的步骤概述

创建一个类一般而言是一个迭代过程

创建一个类的步骤

  • 创建类的总体设计（定义类的职责，精确定义类的抽象概念，指出这个类中关键的公用方法，标识出重要的数据成员）
  • 创建类中的子程序
  • 复查并测试整个类

创建子程序的步骤

  • 设计子程序
  • 检查设计
  • 编写子程序的代码
  • 检查代码

通过伪代码创造子程序

设计子程序

  • 检查先决条件（是否与整体设计匹配，是否真正需要）
  • 定义子程序需要解决的问题（需要隐藏的信息、传给子程序的输入、输出、前置条件、后置条件）
  • 为子程序命名（清晰无歧义）
  • 决定如何测试子程序
  • 在标准库中搜索可用的功能（重用好代码）
  • 考虑错误处理
  • 考虑效率问题
  • 研究算法与数据结构
  • 编写伪代码
  • 考虑数据
  • 检查伪代码
  • 在伪代码中试验一些想法，留下最好的想法

编写子程序

  • 写出子程序的声明
  • 将伪代码转换为更高级别的注释
  • 每条注释下面填充代码

检查子程序

  • 编译子程序
  • 测试子程序
  • 消除代码中的错误

收尾工作

  • 检查设计质量
  • 检查子程序的变量
  • 语句和逻辑
  • 代码风格
  • 子程序文档
  • 去除冗余注释

伪代码编程的代替方案

  • 测试先行开发
  • 重构（不断重构，保持语气的前提下优化代码）
  • 契约式设计（前条件和后条件）
  • 东拼西凑（写出能够工作的代码）
