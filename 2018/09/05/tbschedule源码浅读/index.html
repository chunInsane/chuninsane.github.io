tbschedule源码浅读

Sep 5, 2018

1. overview

远古分布式调度框架，支持分片任务，在taobao生产环境有大规模使用。目前外网公开的源码版本较老，整个项目代码量比较小，适合直接通过源码理解作者的设计

1.1 名词解释

  • clientId：客户端唯一标识，生成规则：ip$hostname$uuid$
  • serverId：处理器（线程组）唯一标识，生成规则：taskType$ip$uuid$
  • ownSign：环境标识，比如：dev、test
  • 任务配置：用于配置任务处理的bean、心跳、线程组线程数、执行时间、终止时间、任务项等数据
  • 调度策略配置：用于配置任务执行的机器列表、线程组数量、任务参数等数据，与任务配置一一映射

2. 设计

2.1 架构设计

  • console：用于查看管理策略、任务类型、任务分配
  • zookeeper：被用作nosql，存储策略、任务类型等数据
  • client：以jar包形式接入到使用方应用

2.2 zk结构

3. 源码剖析

3.1 客户端初始化

3.1.1 创建zk连接，启动初始化线程（TBScheduleManagerFactory#init）

3.1.2 客户端注册，启动管理任务（TBScheduleManagerFactory#initialData）

3.2 动态更新策略任务

3.2.1 重新分配（TBScheduleManagerFactory#reRegisterManagerFactory）

3.2.2 更新策略任务的运行数量（TBScheduleManagerFactory#reRunScheduleServer）

3.3 任务调度管理器

3.3.1 创建任务调度管理器，映射策略任务（TBScheduleManager#new）

3.3.2 初始任务调度管理器（TBScheduleManager#initial）

3.4 任务执行

3.4.1 使用timer实现任务的定时触发（TBScheduleManager#computerStart）

3.4.2 任务执行入口（PauseOrResumeScheduleTask#run）

3.4.3 创建并启动任务执行器（TBScheduleManager#resume）

3.5 任务执行器

3.5.1 NotSleep模式（TBScheduleProcessorNotSleep）

简介：线程从任务队列拉取数据，执行线程发现队列为空则主动拉取数据。拉取数据时可能还有部分任务在其他线程中运行，拉取数据后需要进行去重处理，保证任务不被重复处理。此模式无线程休眠，提高吞吐量的同时会增加cpu消耗（频繁线程上下文切换、数据唯一校验）

3.5.2 Sleep模式（TBScheduleProcessorSleep）

简介：线程从任务队列拉取数据，线程发现队列为空但自己不是最后一个激活状态的线程时进入休眠状态。如果线程发现任务队列为空且自己为最后一个激活状态线程，主动拉取任务数据并添加到任务队列。此场景拉取任务之前任务队列的任务已处理完毕，不存在任务重复被添加到队列的可能

4. 总结

理解了任务类型与策略的关系后，整理的源码阅读体验还是比较流畅，看完发现完全把zk当nosql用了，有点骚~^-^~。任务分配是个过程，如何保证分配过程中的脏数据不被client读到，这些小细节上的设计值得学习。tbschedule整理的设计挺不错，各种场景上的使用也均考虑到了，源码适合进阶学习。
